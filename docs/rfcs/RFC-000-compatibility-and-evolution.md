# RFC-000：兼容性、迁移与演进政策

- 状态：Accepted
- 日期：2026-07-22

## 决策

KineWeave 把架构决定视为可修正的工程判断。早期允许快速、直接地修正错误设计，但不允许通过静默解释旧数据或无限叠加兼容分支来掩盖错误。

### 代码 API

- `Internal` 和 `Experimental` API 可以直接破坏性修改。
- 对 Experimental API 的调用方必须跟随当前版本，不提供旧签名转发层。
- 只有进入 `Stable` 的 API 才承担向后兼容义务。
- Stable API 的废弃必须提供替代路径、弃用期和 Conformance Test。

### 开发期持久化数据（同样不兼容旧草稿）

- Project、Document、Extension-owned Data、Lockfile、Commit/Patch Pack 分别保留版本字段，读取方不得猜测缺失版本。
- 第一次公开格式基线前，只支持仓库当前版本；旧开发工程、Fixture 和样例直接重建。
- 不为未发布草稿提供 Reader、Migration、只读兼容或旧格式回写。
- 迁移代码只服务真实发布过的数据版本；仓库尚未发布的草稿格式直接替换，不建立迁移链。
- 每次破坏性格式修改必须在同一变更中更新 Schema、写入端、读取端、测试 Fixture 和仓库样例，不能以“双格式暂时共存”为过渡方案。

### 公开基线之后

- 可迁移版本先生成迁移计划、诊断和备份。
- 无迁移路径时以只读或 Safe Mode 打开，原始数据必须可导出和无损回写。
- 有损迁移必须显式说明，不能混入普通保存。

### 跨进程协议

- 消息包含协议族、主版本、消息类型和消息版本。
- 握手阶段协商能力；不兼容时明确拒绝，不做隐式字段猜测。
- 同一主版本内允许增加可选字段；删除或改变语义需要新主版本。

### Lockfile

Lockfile 记录精确 Extension、Capability Provider、Runtime 和资源哈希。Manifest 表达需求，Lockfile 表达一次已解析结果。两者不可混为同一职责。

## 为什么不是“先兼容起来”

兼容层只应用于已经对外发布并存在真实工程依赖的 Stable 契约。对仓库内部草稿保留兼容会放大错误边界、增加测试矩阵，并阻碍正确重构，因此明确禁止。
